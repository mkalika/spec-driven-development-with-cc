THREE ADVERSARIAL REVIEWERS · IN PARALLEL

Orthogonal lenses. Same code. Different blind spots.

🎯 Blind Hunter

Reads only the diff. No story context. Treats every line as suspicious.

Finds: race conditions · stale closures · wrong-type comparisons · shared mutable state by reference.

Real catch: a primary-key collision in a crash-recovery edge case. Proved an architectural invariant.

🌐 Edge Case Hunter

Walks every branch and boundary. Empty, null, very large, very small.

Finds: off-by-one at boundaries · 10× scale assumptions that broke at real volume · empty arrays crashing reducers.

Real catch: a budget cap declared as a function-local var. Reset every call instead of every batch.

📋 Acceptance Auditor

Reads spec ACs vs implementation.

Asks: "AC says replay-safe. Does the code actually replay safely?"

Real catch: caught negative-case AC violations across three stories in one epic.

Production ratio: ~30% signal, ~70% noise. Both matter. False-positive triage is required discipline — applying a fix for a false positive has introduced real bugs in this project.

## Mandatory Protocols

### Bug Diagnosis Protocol (MANDATORY)
When a user reports unexpected behavior, run the 5-step protocol BEFORE offering any theory:
1. What did the user actually do? (button/sequence)
2. What handler runs? Find it in code.
3. What state changes? Compute actual values.
4. What does the new state produce visually?
5. Does step 4 match what should happen?
Skipping this protocol is a hard failure.

### Visual Change Protocol (MANDATORY)
When editing any visual/rendering code:
1. BEFORE: take a baseline screenshot.
2. Make the code change.
3. AFTER: take another screenshot of the same view.
4. Compare and describe specific elements.
5. Test at multiple states (3+ different configs).
Skipping this protocol is a hard failure.

### Verify Before Declaring (MANDATORY)
Before stating something "works," provide specific, falsifiable evidence:
- Visual: "element A at X,Y, element B 15px below — they don't overlap."
- Computed: "expected 108, got 108."
- Behavior: state expectation BEFORE observing, then compare.
Skipping this protocol is a hard failure.

### Test Plan Execution (MANDATORY)
Writing a test plan is not credit. If you write tests marked CRITICAL,
you must execute them — numerically, visually, or both — and report
the results before marking work complete.

### Prototype Hard Visuals First (MANDATORY)
When building visually complex components, identify the hardest rendering
challenge and build a minimal standalone prototype that proves the
approach works BEFORE building the full system.

# Bug Post-Mortem Protocol

You are performing a structured post-mortem analysis. Follow ALL steps below
rigorously. Do NOT skip any step. Do NOT mark the task complete until Phase 5
is verified.

## Phase 1: Root Cause Analysis (5 Whys + Causal Chain)

Map the complete failure chain:

  Trigger → [what initiated the failure]
     ↓
  Propagation → [how it moved through the system]
     ↓
  Manifestation → [what the user/system observed]
     ↓
  Root Cause → [the deepest systemic reason this was possible]

Apply iterative "Why?" analysis (minimum 3 levels, target 5).
Stop when you reach a cause you can ACT on.

Classify the bug type:
- Logic error · State management · Contract violation · Boundary failure
- Configuration · Resource · Dependency · Data · UI/UX · Platform

## Phase 2: Detection Gap Analysis

For EACH detection layer, explicitly state whether it SHOULD have caught
this and WHY it didn't:

  | Layer                           | Should catch? | Why missed |
  |---------------------------------|---------------|------------|
  | Type system / compiler          |               |            |
  | Linting / static analysis       |               |            |
  | Unit tests                      |               |            |
  | Integration tests               |               |            |
  | E2E / visual / contract tests   |               |            |
  | Code review                     |               |            |
  | Build / CI pipeline             |               |            |
  | Runtime error tracking          |               |            |

Identify the EARLIEST possible detection point. That's where prevention goes.

Common detection gap patterns:
- Tests used mocks that masked real behavior
- Happy-path-only testing — no negative/boundary cases
- Test environment differs from real usage
- Assertion checked the wrong thing
- No test existed for this code path
- Code review missed it because the bug is subtle
- Error was silently swallowed
- The bug only manifests with specific data/timing/sequence
- Build/bundle difference from dev environment masked the behavior

## Phase 3: Contradiction Analysis

Does fixing this bug risk degrading another quality (performance,
readability, bundle size, UX)? Seek a resolution that improves both
sides rather than accepting a tradeoff.

Examples:
- Validation hurts performance → use compile-time validation
- Tests increase maintenance → generate tests from a schema
- Guard clutters the code → extract an abstraction

## Phase 4: Preventive Implementation (MANDATORY — NOT OPTIONAL)

Implement ALL applicable items. Check each as complete:

### 4a. Regression Test (ALWAYS required)
- Reproduces the EXACT original bug scenario.
- Verify it FAILS without your fix (revert temporarily).
- Verify it PASSES with your fix.

### 4b. Boundary & Edge Case Tests (ALWAYS required)
- Identify at least 2 adjacent failure modes.
- Write tests for those adjacent cases.

### 4c. Type / Contract Hardening (when applicable)
- Tighten types (strict mode, branded types, discriminated unions).
- Add runtime validation at boundaries.

### 4d. Static Analysis Rules (when applicable)
- If the bug follows a detectable pattern, add a lint rule.

### 4e. Defensive Guards (when applicable)
- Add fail-fast assertions for "impossible" states.

### 4f. Observability (when applicable)
- Add structured log/error tracking at the failure point.

### 4g. Documentation (CRITICAL — feeds Layer 1)
- Add inline comment at fix site: `// FIX: [context] — [what went wrong]`
- If this is a new bug pattern for this project, append to
  "Known Bug Patterns" in CLAUDE.md:

  - **[Pattern name]**: [Description]. Prevention: [what catches it].
    First seen: [date/context].

## Phase 5: Verification Checklist

Before declaring the fix complete, verify ALL:
- [ ] Regression test fails without the fix (confirmed by running it)
- [ ] Regression test passes with the fix (confirmed by running it)
- [ ] All new edge case tests pass
- [ ] Full test suite passes (no regressions introduced)
- [ ] Fix addresses root cause, not just symptom
- [ ] At least one preventive measure from Phase 4 has been implemented
- [ ] The project still builds/bundles/deploys cleanly

## Output Format

After completing all phases, provide a structured summary:

  **Bug**: [one-line description]
  **Root Cause**: [the actionable root cause from Phase 1]
  **Bug Type**: [classification from Phase 1]
  **Earliest Detection Point**: [from Phase 2]
  **Detection Gap**: [why the earliest detection point missed it]
  **Preventive Actions Implemented**: [list with file:line references]
  **Contradictions Resolved**: [if any, from Phase 3]
  **Remaining Risk**: [any residual risk]

## 2. Pre-Flight

Read `{{paths.sprint_status}}` once.

For each story_id:
- Resolve full `story_key` from sprint-status (e.g., `12.7` → `12-7-state-manager-sqlite-integration`)
- Validate: status is `backlog` (fresh start) OR `ready-for-dev`/`review` (resume)
- Validate: story's epic is `in-progress` OR `backlog`. If `backlog`, the orchestrator will transition it to `in-progress` automatically — on a live run write the change to sprint-status before Phase A fires; on a dry-run surface it in the plan as "would transition epic-N: backlog → in-progress". Epic status of `done` or `cancelled` is a hard block.

Resolve per-story metadata from config:
- `arch_files[]` — from per-epic architecture allowlist
- `prior_story_files[]` — computed from sprint-status using the config's prior-story rule
- `dev_skill` — config default OR per-story override (may be `skip`)
- `e2e_command` — config default OR per-story override
- `hard_stop_reason` — from per-story hard-stops section, may be empty

Resolve sequence-level metadata:
- `phase_e_enabled` — config flag, AND-ed with `!skip_visual` (the CLI flag forces it off for this run)
- `phase_e_invariants_union` — union of per-epic invariants for all epics touched
- `phase_e_method` — from config (`chrome-devtools-mcp` / `playwright` / `skip`)

**Concern classification (apply before surfacing anything as a pre-flight blocker):**

Before flagging any concern, ask: *would this block story 1's Phase A right now, or would it resolve naturally as the sequence advances?* Classify each as:
- `blocks-at-start` — prevents the first story from running cleanly (missing config section, story_id not found in sprint-status, story already `done`, epic `done`/`cancelled`, missing architecture files, etc.). **These belong in the pre-flight halt list.**
- `resolves-during-sequence` — would close naturally as later stories ship (e.g., Phase E invariant scripts for UI surfaces that don't exist yet, prior-story files that will be authored mid-sequence by earlier stories). **These do NOT belong in pre-flight.** Surface them inside the Phase E plan section as informational notes, not blockers. Unauthored Phase E scripts specifically are caught by the section 4a gate just before Phase E fires — the right moment, after the surfaces under test exist.

The whole point of running Phase E once at sequence end is that the surfaces it tests are built during the sequence — treating their absence today as a pre-flight blocker is a category error.

## Subagent Contract (orchestrator → subagents)

The pipeline runs each phase in a fresh subagent context. To avoid step-on-step state corruption, the orchestrator MUST include these constraints in every Phase A/B/C subagent prompt:

1. **Phase C reviewers MUST NOT modify `sprint_status.yaml` or the story's `Status:` line.** Status transitions belong to the orchestrator alone, which advances them only after the corresponding gate passes. A reviewer that flips a story to `done` before Phase D runs creates a false green. **Witnessed 2026-05-11**: the Phase C reviewer for Story 13.5 wrote `Status: done` to both the story file and sprint-status before Phase D ran. The orchestrator had to verify the gate manually. (Phase A `bmad-create-story` *does* legitimately update sprint-status — `backlog → ready-for-dev` — and Phase B `bmad-dev-story` legitimately advances `ready-for-dev → review` after the unit gate. Only Phase C is constrained.)

2. **Do NOT touch other stories' files.** A Phase B/C subagent for story X may read prior-story files but may not edit them. Cross-story edits indicate the subagent has lost scope.

3. **Return JSON only.** Subagents that emit prose around their JSON force the orchestrator to parse loose markdown.

4. **Patch loop is bounded by `unit_tests`.** Apply patches → run unit tests → if green commit, if red HALT. Subagents that loop on red (re-trying patches against failing tests) should HALT and return `ok: false` with the test log.

These constraints are the orchestrator's responsibility to surface in every prompt — they are not enforceable from the config file alone. If a subagent violates one, log the violation and treat its output as suspect.

# === hooks/session-start.py ===
# Runs at the start of every Claude Code session.
# Injects the knowledge-base index and the latest daily log into context.

"""
SessionStart hook - injects knowledge base context into every conversation.

This is the "context injection" layer. When Claude Code starts a session,
this hook reads the knowledge base index and recent daily log, then injects
them as additional context so Claude always "remembers" what it has learned.

Configure in .claude/settings.json:
{
    "hooks": {
        "SessionStart": [{
            "matcher": "",
            "command": "uv run python hooks/session-start.py"
        }]
    }
}
"""

import json
import sys
from datetime import datetime, timedelta, timezone
from pathlib import Path

# Paths relative to project root
ROOT = Path(__file__).resolve().parent.parent
KNOWLEDGE_DIR = ROOT / "knowledge"
DAILY_DIR = ROOT / "daily"
INDEX_FILE = KNOWLEDGE_DIR / "index.md"

MAX_CONTEXT_CHARS = 20_000
MAX_LOG_LINES = 30


def get_recent_log() -> str:
    """Read the most recent daily log (today or yesterday)."""
    today = datetime.now(timezone.utc).astimezone()

    for offset in range(2):
        date = today - timedelta(days=offset)
        log_path = DAILY_DIR / f"{date.strftime('%Y-%m-%d')}.md"
        if log_path.exists():
            lines = log_path.read_text(encoding="utf-8").splitlines()
            # Return last N lines to keep context small
            recent = lines[-MAX_LOG_LINES:] if len(lines) > MAX_LOG_LINES else lines
            return "\n".join(recent)

    return "(no recent daily log)"


def build_context() -> str:
    """Assemble the context to inject into the conversation."""
    parts = []

    # Today's date
    today = datetime.now(timezone.utc).astimezone()
    parts.append(f"## Today\n{today.strftime('%A, %B %d, %Y')}")

    # Knowledge base index (the core retrieval mechanism)
    if INDEX_FILE.exists():
        index_content = INDEX_FILE.read_text(encoding="utf-8")
        parts.append(f"## Knowledge Base Index\n\n{index_content}")
    else:
        parts.append("## Knowledge Base Index\n\n(empty - no articles compiled yet)")

    # Recent daily log
    recent_log = get_recent_log()
    parts.append(f"## Recent Daily Log\n\n{recent_log}")

    context = "\n\n---\n\n".join(parts)

    # Truncate if too long
    if len(context) > MAX_CONTEXT_CHARS:
        context = context[:MAX_CONTEXT_CHARS] + "\n\n...(truncated)"

    return context


def main():
    context = build_context()

    output = {
        "hookSpecificOutput": {
            "hookEventName": "SessionStart",
            "additionalContext": context,
        }
    }

    print(json.dumps(output))


if __name__ == "__main__":
    main()


# === hooks/pre-compact.py ===
# Runs before Claude Code auto-summarizes a long conversation.
# Captures session state so nothing is lost in compaction.

"""
PreCompact hook - captures conversation transcript before auto-compaction.

When Claude Code's context window fills up, it auto-compacts (summarizes and
discards detail). This hook fires BEFORE that happens, extracting conversation
context and spawning flush.py to extract knowledge that would otherwise
be lost to summarization.

The hook itself does NO API calls - only local file I/O for speed (<10s).
"""

from __future__ import annotations

import json
import logging
import os
import re
import subprocess
import sys
from datetime import datetime, timezone
from pathlib import Path

# Recursion guard
if os.environ.get("CLAUDE_INVOKED_BY"):
    sys.exit(0)

ROOT = Path(__file__).resolve().parent.parent
SCRIPTS_DIR = ROOT / "scripts"
STATE_DIR = SCRIPTS_DIR

logging.basicConfig(
    filename=str(SCRIPTS_DIR / "flush.log"),
    level=logging.INFO,
    format="%(asctime)s %(levelname)s [pre-compact] %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)

MAX_TURNS = 30
MAX_CONTEXT_CHARS = 15_000
MIN_TURNS_TO_FLUSH = 5


def extract_conversation_context(transcript_path: Path) -> tuple[str, int]:
    """Read JSONL transcript and extract last ~N conversation turns as markdown."""
    turns: list[str] = []

    with open(transcript_path, encoding="utf-8") as f:
        for line in f:
            line = line.strip()
            if not line:
                continue
            try:
                entry = json.loads(line)
            except json.JSONDecodeError:
                continue

            msg = entry.get("message", {})
            if isinstance(msg, dict):
                role = msg.get("role", "")
                content = msg.get("content", "")
            else:
                role = entry.get("role", "")
                content = entry.get("content", "")

            if role not in ("user", "assistant"):
                continue

            if isinstance(content, list):
                text_parts = []
                for block in content:
                    if isinstance(block, dict) and block.get("type") == "text":
                        text_parts.append(block.get("text", ""))
                    elif isinstance(block, str):
                        text_parts.append(block)
                content = "\n".join(text_parts)

            if isinstance(content, str) and content.strip():
                label = "User" if role == "user" else "Assistant"
                turns.append(f"**{label}:** {content.strip()}\n")

    recent = turns[-MAX_TURNS:]
    context = "\n".join(recent)

    if len(context) > MAX_CONTEXT_CHARS:
        context = context[-MAX_CONTEXT_CHARS:]
        boundary = context.find("\n**")
        if boundary > 0:
            context = context[boundary + 1 :]

    return context, len(recent)


def main() -> None:
    # Read hook input from stdin
    try:
        raw_input = sys.stdin.read()
        try:
            hook_input: dict = json.loads(raw_input)
        except json.JSONDecodeError:
            fixed_input = re.sub(r'(?<!\\)\\(?!["\\])', r'\\\\', raw_input)
            hook_input = json.loads(fixed_input)
    except (json.JSONDecodeError, ValueError, EOFError) as e:
        logging.error("Failed to parse stdin: %s", e)
        return

    session_id = hook_input.get("session_id", "unknown")
    transcript_path_str = hook_input.get("transcript_path", "")

    logging.info("PreCompact fired: session=%s", session_id)

    # transcript_path can be empty (known Claude Code bug #13668)
    if not transcript_path_str or not isinstance(transcript_path_str, str):
        logging.info("SKIP: no transcript path")
        return

    transcript_path = Path(transcript_path_str)
    if not transcript_path.exists():
        logging.info("SKIP: transcript missing: %s", transcript_path_str)
        return

    # Extract conversation context in the hook
    try:
        context, turn_count = extract_conversation_context(transcript_path)
    except Exception as e:
        logging.error("Context extraction failed: %s", e)
        return

    if not context.strip():
        logging.info("SKIP: empty context")
        return

    if turn_count < MIN_TURNS_TO_FLUSH:
        logging.info("SKIP: only %d turns (min %d)", turn_count, MIN_TURNS_TO_FLUSH)
        return

    # Write context to a temp file for the background process
    timestamp = datetime.now(timezone.utc).astimezone().strftime("%Y%m%d-%H%M%S")
    context_file = STATE_DIR / f"flush-context-{session_id}-{timestamp}.md"
    context_file.write_text(context, encoding="utf-8")

    # Spawn flush.py as a background process
    flush_script = SCRIPTS_DIR / "flush.py"

    cmd = [
        "uv",
        "run",
        "--directory",
        str(ROOT),
        "python",
        str(flush_script),
        str(context_file),
        session_id,
    ]

    creation_flags = subprocess.CREATE_NO_WINDOW if sys.platform == "win32" else 0

    try:
        subprocess.Popen(
            cmd,
            stdout=subprocess.DEVNULL,
            stderr=subprocess.DEVNULL,
            creationflags=creation_flags,
        )
        logging.info("Spawned flush.py for session %s (%d turns, %d chars)", session_id, turn_count, len(context))
    except Exception as e:
        logging.error("Failed to spawn flush.py: %s", e)


if __name__ == "__main__":
    main()

def _summarize_tool_use(block: dict) -> str | None:

    name = block.get("name", "")

8 · Q&A

Twelve questions to expect, with concrete answers. Generic — none require knowing the specific projects in the talk.

Q: This looks like a lot of overhead for small features. Is it worth it?

It is, until it isn't. The discipline isn't about big features — it's about not paying the same lesson twice. A 1-hour retro that saves one re-debug pays for itself.

Concrete example: a shared-state bug that returned a default object by reference. Tests passed in isolation. Production rotted over hours. Without the retro habit, this same bug pattern would reappear in three different forms. With the retro habit, one rule ("never return module defaults by reference") catches it forever.

Q: What's the killer customization to start with?

CLAUDE.md with five rules and the SessionStart hook. Everything else can wait.

Suggested first five rules:

  • "Real-data smoke test before declaring done. Mocks don't count."
  • "Show me file:line for every claim. No theorizing."
  • "Minimal fix first. If you want to refactor more, ask."
  • "State expected value, state actual value, compare. No 'looks right.'"
  • "Code comments are intentional. Don't 'fix' them without asking."

Those will pay for themselves in one week.

Q: How do you decide what becomes a CLAUDE.md rule vs. a knowledge concept?

Rules are for things the AI must check every time. Concepts are for things you might want to look up after a similar incident. If you find yourself adding the same context manually in three sessions, it's a rule.

CLAUDE.md rule examples (always-on, short, imperative):

  • "Never mock external APIs in tests. Always run one live smoke test before declaring done."
  • "Strict TypeScript: no any, no @ts-ignore, no as any casts."
  • "Renderer never touches the filesystem directly — all I/O goes through the IPC layer."

Knowledge concept examples (referenceable, longer, contextual):

  • concepts/shared-state-bugs-in-module-defaults.md — full incident write-up: how a default was returned by reference, the production rot it caused, the fix, related concepts.
  • concepts/scale-broke-an-O(N²)-rule.md — what the rule was, when it stopped working, what changed.
  • concepts/three-layer-defense-explained.md — the framework itself, with links to where each layer lives.

The deciding test: if the AI must act differently because of it → rule. If you'd want to look it up next time something similar happens → concept.

Q: Doesn't this slow you down?

Counterintuitively, no — after the first project setup. The pattern compounds: rules accumulate, the system gets harder to fool, the same bugs stop reappearing.

Concrete pattern: first project takes ~2 hours to set up (CLAUDE.md, hooks, basic pipeline). The first epic is slower than usual because you're learning the cadence. By the 5th epic, velocity is higher than before — because every prior lesson is now automatic.

Q: What does this cost? Time and money.

Money: roughly $1–3 per active Claude Code session for typical API usage. Across five weeks of intense work, total spend in the low three figures.

Time:

  • Setup: ~2 hours one-time (CLAUDE.md + hooks + folder structure)
  • Per-retro: ~30 minutes
  • Across 11 retros over 5 weeks: ~5.5 hours total

Compared against the cost of re-debugging the same class of bug five times, the math is one-sided.

Q: What happens when the model doesn't respect a rule?

It happens. The fix is structural enforcement, not stricter prose.

Concrete example: a rule that said "never delete user data without confirmation" was getting silently ignored — the AI would happily delete entries during a "cleanup" if asked. The fix wasn't to rephrase the rule. The fix was: make the cleanup script refuse to run without a --confirm flag, and put the dry-run output behind a manual gate. Now the rule is enforced by the script, not by hope.

Pattern: if a rule keeps getting violated, make the system physically incapable of violating it. Don't trust prose.

Q: How do you handle the noise from adversarial code review?

Triage discipline. Applying a fix for a false positive can introduce real bugs. The signal that survives includes catches no other gate finds.

Concrete example: an adversarial reviewer flagged a primary-key collision in a crash-recovery edge case — turned out to be a real bug that would have corrupted state on the second crash. No test would have caught it (you can't unit-test "the process died here"). Same review cycle also produced four false-positive flags about formatting. Patch ratio: ~1 real catch per 5 flags. You triage by checking the suspicious-sounding ones first, ignoring style-only flags entirely.

Q: We don't use BMAD or any specific agent framework. Does this work without it?

Yes. The pattern is what matters, not the implementation.

The pattern: spec → dev → review → manual gate, with rules accumulating in a project file. You can run this with any agent harness: Cursor, Aider, plain Claude Code, even a Slack bot. The shape is identical:

  • A spec the AI can be wrong against
  • A review that checks the work adversarially
  • A human gate before "done"
  • A captured lesson if anything slipped
Q: Does this scale to a team, or is it a solo workflow?

It scales better on a team — but the team has to share the artifacts.

What scales: CLAUDE.md (checked into git, shared); the pipeline config (shared); the knowledge folder (shared); the retro habit (shared).

What doesn't scale automatically: the discipline. If only half the team enforces the spec-first habit, the system rots from the loose end. The fix is: make the rules structural — pipeline that won't proceed without a spec, CI that won't merge without manual gate sign-off.

One-team rule: if a rule isn't in CLAUDE.md, it doesn't exist. Verbal agreement decays in a week.

Q: Does this only work with Claude Code, or is it model-agnostic?

The pattern is model-agnostic. The tooling referenced here (CLAUDE.md, hooks, slash commands) is Claude Code-specific.

For other agents:

  • Cursor: use .cursorrules instead of CLAUDE.md. No native hooks, so you maintain the knowledge folder manually.
  • Aider: use .aider.conf.yml + a system prompt file. Hooks become git pre-commit hooks.
  • Cline / Continue: system prompt + rules file work similarly.
  • Plain LLM API: include the rules in every system prompt. No retro automation, but the discipline applies.
Q: How big does CLAUDE.md get before it becomes unwieldy?

In practice, 50–80 rules across 5 categories is the sweet spot. Past 100 rules, the AI starts dropping context.

How to keep it sized:

  • Group rules by category (architecture · testing · AI integration · process)
  • Promote rules to structural enforcement whenever possible (a lint rule, a pre-commit hook, a CI check) and remove them from CLAUDE.md once enforced
  • Demote stale rules to the knowledge folder if they no longer apply
  • Audit every quarter: do all rules still earn their place?

CLAUDE.md is a living document. Rules graduate to code or retire to the archive. They don't sit there forever.

Q: What if the team doesn't buy in? Can one engineer do this alone?

Yes. Start by yourself. Show the results. Adoption follows evidence.

The solo playbook:

  • Week 1: add CLAUDE.md to your own project. Run one Claude Code session through it.
  • Week 2: add the SessionStart hook. Notice the AI is "smarter" because it's loading your accumulated rules.
  • Week 4: you've shipped fewer bugs than your peers. Someone will ask why.
  • Week 6: show them this talk. Or the CLAUDE.md. Or the knowledge folder. Pick whichever resonates.

The team doesn't need to buy in. The team needs to see it work.

Q: What's the one-week challenge?

Pick one Claude Code session this week. Before you start, write down what "done" looks like — something the AI can be measurably wrong against.

At the end, audit: did the AI get away with anything you didn't catch? If yes, write the rule that would have caught it. Put it in your CLAUDE.md.

Start with one rule. Add the next one when the next bug teaches you.
